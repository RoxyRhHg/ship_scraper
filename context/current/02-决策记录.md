## 2026-05-05 — scraper v10 下载失败根因诊断

**诊断结论**（4个根因，按影响大小排序）：

| # | 根因 | 判定 |
|---|------|------|
| 1 | `_discover_all()` 短接逻辑 — bing返回500+候选后 cancel 其他源 | 代码bug |
| 2 | `auto_detect_proxy()` 定义了但从未调用 — `--proxy auto`变成字面量 `"auto"` | 代码bug |
| 3 | `failed_urls` 不持久化 — 每次重启重试死链 | 代码缺失 |
| 4 | 下载异常被 `except Exception` 吞掉 — 无法区分 timeout vs HTTP error vs DNS | 代码缺失 |

**网络判定**：**不是网络问题。** 系统代理 127.0.0.1:7890 工作正常。navsource.org 完全不可达（DNS解析失败）是环境限制，不是代码bug。

**证据**：
- 5/5 测试URL全部下载成功（navysite.de, flickr, alamy, weaponsystems.net, seaforces.org）
- maritimequest.com 直连 + 代理均可达（200）
- wikimedia 通过代理可达，直连 SSL 握手超时（中国网络环境）
- navsource.org DNS 解析失败

## 2026-05-05 — scraper v10 修复（4处改动）

**决策**：在 `ship_scraper_v10/scraper_v10.py` 上做4处修复。

**具体改动**：

1. **移除发现阶段短接** (`_discover_all()`, line ~1588)：删除 `if src_name in ("bing", "usni") and len(all_cands) >= 500: cancel others` 逻辑。bing候选虽多但重复率高，maritimequest/wikimedia才是高价值来源。

2. **修复代理自动检测** (`__init__`, line ~1429)：当 `config.proxy == "auto"` 时调用 `auto_detect_proxy()` 解析为实际代理URL。修复前 `_resolved_proxy = "auto"`（字面量字符串），导致 `proxy_pool` 用无效URL。

3. **持久化 failed_urls** (`_save_state` / `_load_state`)：state JSON 中新增 `failed_urls` 字段，重启时加载已失败的URL集合，不再重试死链。

4. **细分下载失败原因** (`_download_and_prefilter`, line ~1684)：用 `download_failed_{ExceptionType}` 键分别统计不同异常类型（如 `download_failed_Timeout`、`download_failed_ConnectionError`）。

**替代方案考虑**：
- 回退 v9.5 → 拒绝，v9.5 有同样的短接bug
- 完全重写下载层改用 aiohttp → 拒绝，改动太大，先修复核心逻辑问题

**后果**：
- 需要重新运行才能看到效果
- failed_urls 需要一次完整运行才能积累
- maritimequest BFS 爬取可能较慢（走代理时）

## 2026-05-05 — 保持系统代理（trust_env=True）

**决策**：直连池保持 `trust_env=True`，不显式禁用系统代理。

**原因**：用户在中国网络环境，系统代理 (127.0.0.1:7890) 对访问国际站点（maritimequest, wikimedia, bing）是必要的。

**替代方案**：给直连池设 `trust_env=False` → 已测试并回退。wikimedia 直连 SSL 握手超时，maritimequest 直连可用但 BFS 爬取可能更慢。
